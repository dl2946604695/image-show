## 1. 产品概述

光影集是一款面向摄影爱好者、摄影学习者和轻量作品分享用户的图片展示与摄影辅助平台。当前产品已具备图片浏览、分类筛选、搜索、详情查看、登录注册、上传作品、点赞、下载原图和摄影 AI 对话等基础能力。

产品的核心方向不是做一个泛图片仓库，而是做一个“摄影作品展示 + 摄影学习陪伴”的轻社区：用户可以浏览作品、上传作品、通过分类和搜索发现灵感，并向摄影老师 AI 咨询构图、光线、器材、后期等摄影问题。

## 2. 当前项目现状

### 2.1 技术现状

- 前端：React 18、TypeScript、Vite、React Router、Zustand、Tailwind CSS、lucide-react。
- 后端主路径：Cloudflare Pages Functions，位于 `functions/`。
- 数据存储：Cloudflare KV，保存用户、照片、分类和上传图片字节。
- 认证：自研 JWT，密码使用 PBKDF2 哈希。
- AI 对话：`/api/agent` 通过 SSE 流式返回；未配置 `WEGENT_API_KEY` 时使用内置模拟回复。
- 部署：`wrangler.toml` 已配置 Pages 输出目录 `dist` 和 KV binding `PHOTO_STORE`。
- 遗留方案：`backend/`、`api/`、`server.cjs` 等保留了早期 Express / Vercel / 本地服务方案，不是当前主部署路径。

### 2.2 已验证状态

- `npm.cmd run check` 通过。
- `npm.cmd run build` 通过。
- 产物输出到 `dist/`。

### 2.3 已实现产品能力

- 首页摄影画廊。
- 深色视觉风格、固定顶部导航、响应式移动菜单。
- 图片瀑布流展示。
- 分类筛选。
- 关键词搜索，匹配标题、描述和摄影师名称。
- 图片详情弹窗。
- 详情内上一张 / 下一张切换。
- 图片点赞。
- 原图下载。
- 登录 / 注册。
- 登录后上传图片。
- 上传校验：必填标题、分类、图片文件；文件不超过 10MB；文件类型必须为图片。
- 摄影老师 AI 对话页。
- AI 对话支持流式文本输出、停止生成、新建对话、快捷问题卡片。
- 后端种子数据：初始分类、示例照片、默认管理员账号。
- API fallback：前端请求失败或无数据时回退到 mock 数据。

### 2.4 当前不足与占位

- 导航中的部分入口，如精选、摄影师，目前仍指向首页，未形成独立页面。
- 上传页的描述字段有状态和提交逻辑，但当前 UI 未提供描述输入框。
- AI 对话输入区有上传图片按钮，但尚未实现图片上传分析能力。
- AI 历史记录侧栏是静态示例，不是真实会话历史。
- 点赞没有登录限制，也没有用户维度去重，刷新后前端本地 liked 状态会重置。
- 下载原图没有权限策略、版权提示或埋点。
- 用户头像、个人主页、作品管理、删除作品等能力尚未实现。
- 分类目前是固定种子数据，没有后台管理或用户自定义分类。
- 后端 KV 以数组整体读写照片和用户，适合 MVP，不适合高并发或大规模数据。
- README 仍是 Vite 模板内容，部署文档存在编码显示问题，需要整理。

## 3. 产品定位

### 3.1 一句话定位

光影集是一个帮助摄影爱好者展示作品、发现灵感并获得摄影学习建议的轻量图片社区。

### 3.2 核心价值

- 对浏览者：快速发现高质量摄影作品和拍摄灵感。
- 对创作者：低门槛发布作品，形成可浏览的个人作品沉淀。
- 对学习者：通过摄影老师 AI 获得即时、结构化的摄影建议。
- 对产品方：用“作品展示 + AI 学习”形成差异化，而不是只做普通图库。

### 3.3 目标用户

- 摄影新手：想学习构图、曝光、光线和后期技巧。
- 摄影爱好者：想上传和展示自己的摄影作品。
- 灵感浏览者：想按分类或关键词快速浏览图片。
- 内容运营者：未来可管理精选作品、分类和推荐内容。

## 4. 产品目标

### 4.1 MVP 目标

- 用户能稳定浏览作品。
- 用户能通过分类和搜索找到感兴趣作品。
- 用户能查看作品详情并进行基础互动。
- 登录用户能上传图片作品。
- 用户能和摄影老师 AI 进行摄影问答。
- 产品能通过 Cloudflare Pages 完成部署并运行。

### 4.2 下一版本目标

- 补齐作品发布信息：描述、标签、拍摄参数、地点等。
- 实现真实摄影师主页和作品集合。
- 实现精选频道与运营推荐。
- 增强 AI：支持基于用户上传图片的点评与建议。
- 完善用户体系：个人资料、我的作品、删除/编辑作品。
- 改善内容安全与版权合规。

### 4.3 非目标

- MVP 不做复杂社交关系链，如关注、私信、评论楼中楼。
- MVP 不做商业图库交易、授权购买或版权结算。
- MVP 不做专业 DAM 资产管理。
- MVP 不做复杂后台 CMS，除非运营需求明确。
- MVP 不做无限制大容量原图存储承诺。

## 7. 功能需求

### 7.1 首页画廊

#### 需求描述

首页是产品第一入口，负责展示平台作品、搜索和分类筛选结果。

#### 当前能力

- 页面顶部显示品牌、导航、搜索框和登录/上传入口。
- 首页 hero 展示产品气质、作品数量和分类数量。
- 图片以瀑布流展示。
- 加载态使用骨架屏。
- 无结果时显示空状态。

#### 需求规则

- 默认按发布时间倒序展示作品。
- 当接口无数据或请求失败时，可使用 mock 数据保障演示体验。
- 分类筛选与关键词搜索可以叠加。
- 搜索范围包括标题、描述、摄影师名称。
- 移动端应保留搜索和导航入口。

#### 验收标准

- 进入 `/` 能看到图片列表。
- 点击分类后，仅展示该分类作品。
- 输入关键词后，列表实时过滤。
- 分类和搜索同时生效。
- 无匹配结果时显示空状态。
- 接口失败时页面不崩溃。

### 7.2 图片卡片

#### 需求描述

图片卡片用于展示作品缩略图、分类和基本作者信息。

#### 当前能力

- 使用缩略图懒加载。
- 鼠标悬停有视觉反馈。
- 卡片显示分类标签、标题和摄影师名称。
- 点击卡片打开详情弹窗。

#### 验收标准

- 图片加载前后布局稳定。
- 点击任意卡片打开对应作品详情。
- 图片 alt 使用作品标题。
- 缩略图失败时应有可接受的 fallback 策略。

### 7.3 图片详情

#### 需求描述

详情弹窗提供沉浸式看图体验和基础互动。

#### 当前能力

- 大图展示。
- 图片加载态。
- 图片加载失败时回退到缩略图。
- 展示标题、描述、作者、分类、创建日期。
- 支持关闭弹窗。
- 支持点击遮罩关闭。
- 支持键盘 Esc 关闭。
- 支持键盘左右键和按钮切换上一张 / 下一张。
- 支持点赞和下载原图。

#### 需求规则

- 点赞按钮点击后应立即乐观更新。
- 原图下载使用图片标题作为默认文件名。
- 图片详情切换时应重置加载态和本地点赞态。
- 详情展示的作品序列应与当前筛选后的列表一致；当前实现基于 store 中全量照片，需要后续确认是否调整。

#### 验收标准

- 点击卡片打开正确作品详情。
- 点击关闭按钮、遮罩或按 Esc 能关闭详情。
- 点击左右按钮能切换作品。
- 按键盘左右键能切换作品。
- 点赞后数字增加。
- 下载按钮能触发浏览器下载。

### 7.4 登录与注册

#### 需求描述

用户通过邮箱、密码和用户名完成注册，使用邮箱密码登录。

#### 当前能力

- 登录和注册在同一页面切换。
- 登录成功后保存 token 到 localStorage。
- 注册成功后自动登录。
- 支持显示/隐藏密码。
- 页面加载时会初始化认证状态。
- 上传页未登录时跳转登录页。

#### 需求规则

- 登录字段：邮箱、密码。
- 注册字段：用户名、邮箱、密码。
- token 默认有效期 7 天。
- 退出登录清除本地 token。
- 生产环境必须配置强 JWT secret。

#### 验收标准

- 新用户可注册成功。
- 已注册用户可登录成功。
- 错误账号或密码返回明确错误。
- 登录后导航栏显示上传入口和用户头像入口。
- 退出登录后上传入口不可用。

### 7.5 作品上传

#### 需求描述

登录用户可以上传图片作品并发布到画廊。

#### 当前能力

- 未登录访问 `/upload` 会跳转登录。
- 支持选择图片文件并预览。
- 校验图片类型。
- 校验文件大小不超过 10MB。
- 提交字段包括标题、描述、分类和文件。
- 上传成功后返回首页。
- 后端将图片字节保存到 KV 的 `img:<id>`。

#### 当前缺口

- 前端 UI 未提供描述输入框，但提交逻辑已包含 description。
- 上传后前端没有强制刷新 store，可能依赖重新进入或接口加载。
- 不支持编辑、删除、重新选择分类管理。

#### 需求规则

- 标题必填。
- 分类必填。
- 图片必填。
- 描述建议选填，但 UI 必须可填写。
- 文件最大 10MB。
- 文件类型必须为 image/*。
- 成功发布后应能在首页看到新作品。

#### 验收标准

- 未登录用户进入上传页会被引导登录。
- 登录用户可选择图片并看到预览。
- 超过 10MB 文件被阻止并展示错误。
- 非图片文件被阻止并展示错误。
- 未填写标题、分类或图片时无法提交。
- 上传成功后跳转首页，作品出现在列表中。

### 7.6 摄影老师 AI

#### 需求描述

摄影老师 AI 为用户提供摄影知识问答和创作建议，是产品差异化模块。

#### 当前能力

- 独立页面 `/agent`。
- 初始页展示欢迎语和快捷问题卡片。
- 支持用户输入文本问题。
- 支持 Enter 发送，Shift + Enter 换行。
- 支持流式接收回复。
- 支持中止当前回复。
- 支持新建对话。
- 支持简单 Markdown 段落、无序列表、有序列表和加粗渲染。
- 无真实 API key 时使用 mock SSE 响应。

#### 当前缺口

- 侧边历史记录为静态数据。
- 图片上传按钮未接入图片分析。
- 无会话持久化。
- 无用户身份绑定。
- 无模型错误、超时、限流的完整 UI。

#### 需求规则

- AI 需优先回答摄影相关问题。
- 非摄影问题可礼貌引导回摄影创作、器材、后期、审美和作品分析。
- 未配置外部 AI key 时，必须保留 mock 响应用于演示。
- 流式返回格式为 `data: {"content":"..."}`，结束为 `data: [DONE]`。

#### 验收标准

- 进入 `/agent` 能看到 AI 对话界面。
- 点击快捷卡片可自动发送对应问题。
- 输入问题点击发送后能看到用户消息和 AI 流式回复。
- 回复中点击按钮可中止生成。
- 点击新建对话清空当前对话。
- 未配置真实模型 key 时仍可得到模拟回复。

### 7.7 导航与信息架构

## 8. 后端与接口需求

### 8.1 API 列表

#### Auth

- `POST /api/auth/signup`
  - 入参：email、password、name。
  - 出参：user、token。

- `POST /api/auth/login`
  - 入参：email、password。
  - 出参：user、token。

#### Photos

- `GET /api/photos`
  - 可选 query：category。
  - 出参：照片数组。

- `GET /api/photos/:id`
  - 出参：单张照片。

- `PUT /api/photos/:id/like`
  - 出参：最新 likes。

- `GET /api/photos/:id/image`
  - 出参：图片二进制。

- `POST /api/photos/upload`
  - 鉴权：需要 Bearer token。
  - 入参：multipart formData，包含 title、description、category、file。
  - 出参：新照片对象。

- `GET /api/photos/categories`
  - 出参：分类数组。

#### Agent

- `POST /api/agent`
  - 入参：message。
  - 出参：SSE stream。

### 8.2 数据模型

#### User

- id：用户 ID。
- email：邮箱。
- name：用户名。
- password：服务端存储哈希值，不返回前端。
- createdAt：创建时间。
- avatarUrl：头像地址，当前前端类型预留。

#### Photo

- id：照片 ID。
- title：标题。
- description：描述。
- url：原图地址。
- thumbnailUrl：缩略图地址。
- category：分类名。
- photographerId：摄影师用户 ID。
- photographerName：摄影师展示名。
- createdAt：发布时间。
- likes：点赞数。

#### Category

- id：分类 ID。
- name：分类名称。
- icon：图标名。

### 8.3 数据存储规则

- MVP 阶段可以继续使用 Cloudflare KV。
- 图片元数据保存在 `app:photos`。
- 分类保存在 `app:categories`。
- 用户保存在 `app:users`。
- 图片二进制保存在 `img:<photoId>`。
- 后续增长阶段应评估迁移到 D1、R2 或 Supabase/Postgres，以支持分页、查询、权限和更大文件量。

## 9. 权限与安全

### 9.1 当前策略

- 浏览照片：无需登录。
- 查看详情：无需登录。
- 点赞：无需登录。
- 下载：无需登录。
- 上传：需要登录。
- AI 对话：无需登录。

### 9.2 待确认策略

- 点赞是否需要登录。
- 下载原图是否需要登录或展示版权提示。
- AI 对话是否需要登录、限次或计费。
- 上传作品是否需要审核。
- 是否允许用户删除自己的作品。

### 9.3 安全要求

- 生产环境必须设置 `JWT_SECRET`。
- 上传文件必须限制大小和类型。
- 前后端都要校验上传参数。
- API 不应返回密码哈希。
- AI key 不得暴露到前端。
- CORS 策略生产环境应收敛到可信域名。

## 10. 内容与合规

### 10.1 上传内容规范

- 禁止上传违法、侵权、色情、暴力、仇恨或明显不适内容。
- 用户应拥有作品版权或授权。
- 平台应在上传前提示用户确认版权责任。

### 10.2 推荐与精选规范

- 精选内容应以摄影质量、主题完整度、视觉表现和社区正向价值为标准。
- 不应仅按点赞排序作为精选依据。

### 10.3 AI 回复规范

- AI 应避免给出危险拍摄建议，例如非法闯入、危险地形冒险、侵犯隐私偷拍。
- AI 对器材建议应根据预算、场景和水平给出分层建议。
- AI 不应承诺替代专业法律、商业版权或人身安全判断。

## 11. 非功能需求

### 11.1 性能

- 首页图片应懒加载。
- 详情大图加载时必须有 loading。
- 图片列表后续应支持分页或无限滚动。
- KV 数组整体读写在作品数量增长后需要替换。

### 11.2 可用性

- 接口失败时页面不崩溃。
- 上传失败要显示明确错误。
- AI 失败要给出可理解 fallback。
- 移动端必须可完成浏览、搜索、登录和上传。

### 11.3 可访问性

- 关键按钮需要 aria-label。
- 图片 alt 使用标题。
- 键盘可关闭详情弹窗。
- 后续需补齐焦点管理，尤其是 modal 打开后的焦点 trap。

### 11.4 可维护性

- README 应替换为真实项目说明。
- 部署文档应修复编码问题。
- 遗留 backend/api/server 方案应标注或清理，避免开发者误用。
- mock 数据应统一来源，避免 `src/lib/api.ts` 与 `src/lib/mockData.ts` 重复维护。

## 12. 版本规划

### 12.1 V1.0 MVP 修整版

目标：让当前已实现功能成为可信可演示版本。

范围：

- 修复上传页描述输入框缺失。
- 上传成功后刷新或更新首页照片列表。
- 隐藏或标注未实现导航入口。
- 完善 README。
- 修复部署文档编码。
- 统一 mock 数据来源。
- 为点赞失败增加回滚或错误提示。
- 增加基础接口和页面测试。

### 12.2 V1.1 内容发现版

目标：让用户更容易发现内容和作者。

范围：

- 精选频道。
- 摄影师列表。
- 摄影师主页。
- 图片列表分页或无限滚动。
- 标签体系。
- 排序：最新、热门、最多点赞。

### 12.3 V1.2 创作者中心版

目标：让创作者能管理自己的作品。

范围：

- 我的作品。
- 编辑作品信息。
- 删除作品。
- 用户资料编辑。
- 头像上传。
- 上传版权确认。

### 12.4 V1.3 AI 点评版

目标：把 AI 从问答工具升级为作品点评助手。

范围：

- AI 对话支持图片上传。
- AI 基于图片给出构图、曝光、色彩、主体、裁切建议。
- 保存 AI 点评历史。
- 将某张作品带入 AI 对话上下文。

## 13. 关键流程

### 13.1 浏览作品流程

1. 用户进入首页。
2. 系统请求照片和分类。
3. 页面展示 hero、分类和瀑布流。
4. 用户点击分类或搜索关键词。
5. 页面实时过滤作品。
6. 用户点击图片打开详情。
7. 用户进行点赞、下载或切换图片。

### 13.2 上传作品流程

1. 用户点击上传。
2. 未登录用户跳转登录；已登录用户进入上传页。
3. 用户选择图片。
4. 前端校验文件类型和大小。
5. 用户填写标题、描述、分类。
6. 用户提交表单。
7. 后端验证 token 和表单。
8. 后端保存图片字节和照片元数据。
9. 前端跳转首页并展示新作品。

### 13.3 AI 问答流程

1. 用户进入摄影老师 AI 页面。
2. 用户点击快捷问题或输入问题。
3. 前端创建用户消息和空 AI 消息。
4. 后端连接真实模型或 mock 回复。
5. 前端按 SSE chunk 追加内容。
6. 用户可中止回复或新建对话。

## 15. 风险与应对

### 15.1 KV 扩展性风险

风险：照片和用户以数组整体存储，数据量变大后读写冲突和性能会恶化。  
应对：MVP 可接受；V1.1 前评估分页数据结构，V1.2 前考虑 D1/R2 或关系型数据库。

### 15.2 上传存储风险

风险：图片字节直接放 KV 不适合大量或大图存储。  
应对：MVP 限制 10MB；后续迁移到 R2 存储原图和缩略图。

### 15.3 内容安全风险

风险：用户上传内容无审核，可能出现侵权或违规内容。  
应对：上线前增加版权确认和举报入口；公开发布前考虑审核机制。

### 15.4 AI 成本与质量风险

风险：真实 AI key 接入后可能产生不可控成本和非摄影回答。  
应对：增加登录限制、频控、主题约束 prompt、错误降级和调用日志。

### 15.5 产品边界风险

风险：同时做图库、社区、AI、作品管理，范围容易膨胀。  
应对：V1.0 聚焦“浏览 + 上传 + AI 问答”闭环；精选、摄影师、AI 点评后移。

## 16. 需要产品确认的问题

1. 光影集的优先定位是“作品展示社区”还是“摄影学习工具”？
2. 未登录用户是否允许点赞？
3. 下载原图是否需要登录、版权提示或禁止下载？
4. 用户上传作品是否默认公开？
5. 是否需要作品审核？
6. 分类是否固定，还是允许后台配置？
7. 精选频道由算法产生、人工运营，还是按点赞排序？
8. 摄影师主页是否必须进入 V1.0？
9. AI 是否只回答摄影问题？
10. AI 图片点评是否进入下一版核心需求？
11. 上传图片是否需要生成缩略图，而不是原图复用？
12. 是否需要保留示例 mock 数据作为线上空状态内容？
13. 默认管理员账号是否只用于本地演示，生产是否禁用？

## 17. Grill-me 审视结论

- 如果只说“图片展示”，产品差异化不足；AI 摄影老师应成为明确卖点。
- 如果只做“AI 问答”，当前画廊和上传又会变成装饰；需要让作品能进入 AI 点评上下文。
- 如果 V1.0 就做完整社区，会拖慢上线；先把当前闭环打磨到可靠更重要。
- 如果不修复上传描述 UI、导航占位和文档问题，演示时会显得不完整。
- 如果继续使用 KV 数组整体存储，短期能跑，长期一定要迁移。
- 如果不处理版权和内容安全，公开发布会有明显风险。
